# 存档系统架构设计指南

> 本指南总结自当前项目（DM42 / 爆战兄弟 2）的存档实现，目的是帮后来的项目搭建一套**可靠、可扩展、可防作弊、可多人同步**的工程级存档架构。  
> 仅讲设计思路、流程、注意点，不涉及具体代码。具体 API 参考 `knowledge/核心系统/04-存档系统.md`（Y3 引擎层接口）。

---

## 1. 设计目标

一个工程级存档系统至少要满足以下 7 项硬指标：

| 目标 | 说明 |
|------|------|
| **可靠** | 任何业务变更不丢数据、不串槽位、不脏读 |
| **高性能** | 读多写少场景下命中内存缓存，不每次穿透到引擎 |
| **可防作弊** | 关键字段加密 + 校验 + 行为审计 |
| **多人一致** | 任一玩家的存档变更，房间内所有客户端视图一致 |
| **可演化** | 新增字段、字段格式变更时，旧存档可平滑迁移 |
| **可调试** | 提供 GM 命令查看、清理、伪造存档 |
| **可参数化** | 槽位号、加密 key、字段名集中定义，禁止散落硬编码 |

---

## 2. 推荐五层架构

```
┌─────────────────────────────────────────────────────────────┐
│ 第 5 层：Manager 业务层                                       │
│   组合 DB 层数据 + 配置表 + 引擎对象，对外暴露业务 API         │
│   （如 装备穿戴、商店刷新、邮件领取、抽卡结算）                 │
├─────────────────────────────────────────────────────────────┤
│ 第 4 层：Sync 同步层                                          │
│   把"所有会改变存档的写操作"包装成同步事件，由房主或服务器仲裁  │
│   每个玩家本地按相同序列重放，保证多人一致                     │
├─────────────────────────────────────────────────────────────┤
│ 第 3 层：DB 数据访问层（核心）                                │
│   每个业务子系统对应一个 DB 类，负责：                         │
│   - 懒加载存档槽位                                            │
│   - 内存缓存                                                  │
│   - 编码 / 解码                                               │
│   - 增删改查接口                                              │
│   - 字段校验（反作弊第一道）                                   │
├─────────────────────────────────────────────────────────────┤
│ 第 2 层：加密 / 反作弊层                                      │
│   对敏感字段做 AES + Base64，结合玩家 UUID 作为加密参数        │
│   提供集中的密钥表与编解码工具                                 │
├─────────────────────────────────────────────────────────────┤
│ 第 1 层：引擎接口层（Y3 提供）                                │
│   y3.save_data.load_table / load_integer / add_integer ...    │
│   archive.json 中声明的槽位 + 类型                            │
└─────────────────────────────────────────────────────────────┘
```

**层间约束（强制）**：

- Manager 层**只能**调 DB 层，禁止直接调 `y3.save_data.*`
- DB 层**只能**调加密层 + 引擎接口层，禁止反向引用 Manager
- Sync 层位于 Manager 之上，做全局事件分发，不做业务计算
- 加密层是公共基础设施，被 DB 层和必要的 Manager 共享，不依赖任何业务

## 3. 制作流程（从 0 到 1，11 步）

### Step 1：盘点业务，列出所有需要持久化的子系统

- 逐项写出：玩家基础数据、货币、关卡、装备、技能、抽卡、邮件、商店、排行榜、签到、设置……
- 每项标注：**数据形态**（KV / 列表 / 嵌套对象）、**容量级别**（数十条 / 上千条）、**写频率**（每秒级 / 每局级 / 每天级）、**是否敏感**（影响战力或货币 → 敏感）

> 这一步决定后续槽位数量与类型选择，是后期最难推翻的决策，宁可多花时间。

### Step 2：规划存档槽位（Slot Plan）

- **每个子系统占 1~3 个槽位**，不要把多个子系统挤进一个槽位（会导致互相污染）
- 排行榜类整数累计字段单独占整数槽位，与表格槽位分开
- **预留 5~10 个空槽位**作为后续扩展，槽位号一旦上线不可改
- 用一个集中的 `SaveDataConst`（或同名常量文件）登记 `slotIdx → name` 映射，**禁止任何地方写裸数字槽位号**
- 同步在 `archive.json` 写明每个槽位的 `type`（STR/INT/FLOAT/BOOL/TABLE）和 `encrypt`

> ⚠️ 槽位是**全局单例资源**，整个项目共享。一旦冲突，后患无穷。

### Step 3：定义统一的 DB 类骨架

每个 DB 类应实现**相同的方法模式**，以便审查和复用：

| 方法 | 职责 |
|------|------|
| `getXxxDB(player)` | 懒加载，首次访问时从引擎读，之后走内存缓存 |
| `getCacheData(playerId, ...)` | 业务侧获取缓存的便捷入口 |
| `getXxxConfig(playerId, id)` | 单条数据读取，含解密 + 解码 |
| `saveXxxConfig(playerId, id, data)` | 单条数据写入，含校验 + 编码 + 加密 |
| `clearXxx(playerId)` | 清空数据（GM/重置场景使用） |

> 推行"DB 类一个模板"，新人加新子系统时照抄即可，错不了大方向。

### Step 4：设计编码格式

存档内每个"逻辑单元"通常落成**一个加密字符串**。两种主流编码方案：

**方案 A：管道分隔（推荐用于固定字段）**  
例：`type|sourceId|status|sendTime|count|seq`  
优点：紧凑、解析快、字段顺序固定可向后兼容（末尾追加字段）  
注意点：解码时**用 `tonumber` 容错**并提供默认值，否则旧存档会崩

**方案 B：JSON 序列化（推荐用于嵌套结构）**  
例：`title#{json对象}`  
优点：结构表达力强  
注意点：JSON 解析失败时必须有降级路径，不能让单个坏字段导致整玩家存档不可读

**方案 C：标题前缀校验（强烈建议）**  
所有写入字符串前加一个固定前缀（如 `Currency`、`eqpSavePage`），读取时如果前缀不匹配就视为篡改并报告。这是反作弊的低成本第一道。

### Step 5：搭建加密层

- 用对称加密（AES）+ Base64 编码，**不要明文写入敏感数据**
- 加密参数（key/iv）：
  - **公共密钥**：用于无玩家上下文的初始化数据
  - **玩家密钥**：从玩家 UUID 派生（标准化为固定长度），保证玩家 A 不能拿玩家 B 的存档解密
- 维护一个集中的**密钥表**，每个敏感字段配一个独立 key（避免一处泄露全线沦陷）
- 编辑器模式下 UUID 可能不稳定 → **首次读取时把 UUID 写入存档，后续以存档内 UUID 为准**，否则存档会读不出来

> ⚠️ 所有加密 key 都是源码里的字符串常量，本质上是**混淆**而非真加密。但只要客户端不被反编译，对常规外挂作者就够了。**真正反作弊还要靠服务器校验**（如果有服务器）。

### Step 6：在 DB 层实现"反作弊三件套"

每次写入时强制做：

1. **id 校验**：写入数据的 `data.id` 必须等于 `dbTitleStr .. id`，不一致 → 视为篡改
2. **UUID 校验**（针对长期持有的物品如装备）：物品诞生时记录 UUID，后续每次读出都校验，UUID 漂移 → 视为篡改
3. **审计上报**：发现篡改记录到 TraceMgr / 服务器日志，包含玩家名、槽位名、异常值，便于事后封号

> 反作弊不是"挡住所有人"，而是"提高作弊成本"+"留下证据"。

### Step 7：实现内存缓存

- 第一次访问玩家槽位时，把**整张表**读进内存，之后所有读写都在缓存上做
- 写入操作要**双写**：内存对象 + 引擎 `y3.save_data` 表代理
- TABLE 类型在 Y3 中以"代理表"形式返回，对其字段赋值会自动同步到存档，**不要 `pcall` 包裹后吞错**，会丢数据
- 玩家断线重连后，缓存失效需重新加载（设计 reload 入口）

### Step 8：搭建同步层

多人模式下，**所有会改变存档的操作都必须走同步事件**，否则会出现"我打的怪扣了别人的金币"或"商店在我这里刷新了在他那里没刷新"。

通用流程：

```
玩家 A 触发 → 本地不直接改存档 → 发同步事件 → 房主/服务器接收 → 广播给所有玩家 → 各客户端在事件回调里调用 DB 写入
```

注意点：

- **同步事件常量**集中在 `SyncEventConst` 中维护，禁止散写字符串
- 同步事件的 payload 要带玩家 ID，让每个客户端知道是给谁改
- 复杂操作要拆成"请求 + 应答"两步，避免半途中断造成不一致

### Step 9：建立"配置表 ↔ 存档"的清晰边界

- 配置表（Excel）= 静态数值，所有玩家相同，不入存档
- 存档 = 玩家的动态进度，**只存"和默认值不同的部分"或者"必须的状态机字段"**
- 存档里**不要存配置表里能查到的字段**（如装备名、技能描述），只存 ID 和等级，渲染时再去查表

> 这能让存档体积稳定可控，配置表更新时不需要迁移存档。

### Step 10：设计调试/GM 工具

至少提供以下 GM 命令：

- `save` - 打印当前玩家某槽位的解密数据
- `clearSave <slot>` - 清空某槽位（测试用）
- `addCurrency / addItem` - 直接修改存档的捷径
- `mockSave` - 伪造旧版本存档，用于回归测试迁移逻辑

> 没有调试工具的存档系统等于黑盒，问题排查成本爆炸。

### Step 11：写迁移逻辑（一定会有）

项目运行半年以上必然遇到"老玩家存档与新版字段不匹配"。常见做法：

- **末尾追加字段**：解码时用 `or 默认值`兜底（最简单，推荐）
- **字段重命名**：保留旧字段读取分支，写入时只写新字段（一个版本周期后删除旧分支）
- **结构重构**：在 PlayerDB 里写一个版本号字段；启动时检测版本号，跑迁移函数后写回新版本号
- **一次性补齐**：用 `playerEventMgr:isUsed("升级标识")` 模式标记某玩家是否已跑过某次迁移，避免重复执行

## 4. 关键设计点（容易踩的坑）

### 4.1 槽位是"工程级单例资源"

槽位号 = 全局唯一标识，**不能模板化、不能复用、不能动态分配**。  
新人加子系统时**必须先到 `SaveDataConst` 申请槽位**，由项目负责人审批分配。

### 4.2 整数槽位与表格槽位严格分开

排行榜累计这种用 `add_integer` 的需求**必须用 INT 槽位**，不能在 TABLE 里塞数字字段。INT 槽位有原子加法，TABLE 没有。

### 4.3 PlayerDB 是"杂货抽屉"，但要克制

工程实践中确实存在一个 PlayerDB，存放装备页、UUID、登录时间、强化等级、宝石、符石等"无主"字段。但要警惕：

- 控制 PlayerDB 内字段数量（建议 ≤ 30 个）
- 字段名必须有前缀（如 `eqpSavePage`、`upgradeEqp`）防冲突
- 一旦某个字段子系统化，立即拆出独立 DB

### 4.4 防"半成品状态"

写入操作往往涉及多个字段（货币 -100 + 物品 +1）。如果中途崩溃，会留下不一致状态。两种解法：

- **乐观法**：先扣货币，再加物品，加物品失败则回滚货币（适合简单场景）
- **悲观法**：先校验所有前置条件 → 全部通过再统一写入 → 任何一步失败整体不动

> Y3 单机/小房间场景多用乐观法，附加幂等性保护（用 `seq` 序列号防重复执行）。

### 4.5 缓存一致性

内存缓存 + 引擎存档代理 + 多个客户端同时读写，三方一致是难点。原则：

- **写入路径单一**：所有写都走 DB 层的某个唯一方法，禁止业务直接改缓存
- **写后立即同步缓存**：DB 写入后立即更新本地业务缓存，不等下次读取
- **跨进程一致性靠同步层**，不要试图在 DB 层解决

### 4.6 玩家离线 vs 游戏内

存档是玩家个人的，**离线玩家无法被读写**。多人模式下要小心：

- 房主操作时不要假设其他玩家在线
- 玩家离开时主动调 `clearXxx(playerId)` 释放缓存
- 重连玩家要**强制重读存档**而非用旧缓存

### 4.7 数据兼容是终身工作

**任何已上线的字段不能改类型，不能改语义，只能"追加新字段并标注废弃旧字段"。**  
如果实在要改，必须出迁移函数 + 测试用例 + 灰度方案。

### 4.8 体积控制

Y3 单玩家存档总大小有上限（具体看引擎版本）。注意：

- 列表类数据（邮件、库存、抽卡历史）必须有**最大长度**，超出按 LRU 或时间淘汰
- 不要存"渲染层中间结果"，渲染时计算
- 大字符串（如玩家自定义名称）要限制长度

## 5. 反作弊机制清单

按防御深度分 5 道：

| 道 | 措施 | 抵抗的攻击 |
|----|------|-----------|
| 1 | 字段加密（AES + 玩家 UUID 派生 key） | 直接修改存档文件 |
| 2 | 标题前缀校验（如 `Currency#...`） | 修改加密后字符串结构 |
| 3 | id 自校验（`data.id == titleStr..id`） | 替换字段 |
| 4 | UUID 自校验（装备类长期物品） | 复制别人的存档片段 |
| 5 | 服务器对账（如有后端） | 全部本地手段 |

每发现一道防御被击穿，写入审计日志（玩家名 + 槽位名 + 异常值），事后人工或自动处置。

---

## 6. 多人同步要点

- **同步事件不可丢、不可重复**：用序列号（seq）+ 应答机制保证
- **房主迁移时**要重新绑定所有事件（参考 `BaoZhanArchivedMgr` 中 `玩家-房主玩家变更`）
- **离线玩家的存档不参与同步**：等他重连或下次开局再处理
- **存档相关 UI** 通过 `gameEventMgr:Publish` 推数据，不直接拉 DB（避免每帧查存档）

## 7. 检查清单（项目上线前过一遍）

- [ ] `SaveDataConst` 集中登记所有槽位，无裸数字
- [ ] `archive.json` 与 `SaveDataConst` 一致，类型/加密标记正确
- [ ] 所有 DB 类遵循统一骨架（懒加载 + 缓存 + 编解码 + 校验）
- [ ] 所有敏感字段走加密层，密钥集中管理
- [ ] 反作弊三件套（id 校验、UUID 校验、审计上报）已就位
- [ ] 所有写操作走同步层，事件常量集中
- [ ] GM 命令至少能：查、清、伪造、迁移测试
- [ ] 列表类数据有上限，老数据有淘汰策略
- [ ] 字段追加流程已建立（不可改类型，只可追加 + 兜底）
- [ ] 玩家离开释放缓存，重连重读
- [ ] 至少跑过 3 个版本的迁移演练

## 8. 反模式（这些做法会让你后悔）

1. **多个子系统共用一个 TABLE 槽位** —— 互相覆盖
2. **裸 `y3.save_data.load_table` 散落各处** —— 没有缓存、没有校验
3. **业务层直接改 TABLE 代理字段，不走 DB 方法** —— 反作弊和审计全部失效
4. **加密 key 写在和数据同文件** —— 等于没加密
5. **同步事件用魔法字符串** —— 后续重命名地狱
6. **存档版本号缺失** —— 迁移时分不清新老
7. **没有 GM 命令** —— 线上出问题只能猜
8. **配置表字段冗余存到存档** —— 改表必须迁移存档
9. **依赖 UUID 自动稳定**（编辑器模式 UUID 会变） —— 编辑器测试时存档读不出
10. **失败操作只 `log` 不上报** —— 作弊行为查无证据

## 附录：各层的文件组织约定

按本指南的五层架构，一个工程级存档系统在项目目录中的组织方式建议：

```
script/
├── const/
│   ├── SaveDataConst.lua       # 槽位号枚举 + 名称映射（唯一真实来源）
│   └── SyncEventConst.lua      # 同步事件名枚举
├── enc/
│   └── EncryptMgr.lua          # 加密层：密钥表 + AES 编解码 + UUID 规范化
├── db/
│   ├── DBMgr.lua               # DB 聚合器，实例化所有子 DB
│   ├── PlayerDB.lua            # 玩家基础数据 DB
│   ├── CurrencyDB.lua          # 货币 DB
│   ├── EquipmentDB.lua         # 装备 DB
│   ├── MailDB.lua              # 邮件 DB
│   └── ...                     # 按子系统逐个建立
├── sync/
│   ├── SyncMgr.lua             # 同步管理器，注册所有事件
│   ├── SyncEvt_Currency.lua    # 货币变更同步
│   ├── SyncEvt_Equipment.lua   # 装备变更同步
│   └── ...                     # 按子系统逐个建立
├── mgr/
│   ├── EquipmentMgr.lua        # 装备业务逻辑
│   ├── MailMgr.lua             # 邮件业务逻辑
│   └── ...                     # 按子系统逐个建立
└── debug/
    └── GM.lua                  # GM 调试命令（查档/清档/伪造/迁移）
```

**命名约定**（建议遵守）：

| 层级 | 后缀 | 示例 |
|------|------|------|
| 常量 | `Const` | `SaveDataConst`、`SyncEventConst` |
| 加密 | `Encrypt` + `Mgr` | `EncryptMgr` |
| 数据访问 | `DB` | `PlayerDB`、`CurrencyDB`、`MailDB` |
| 同步事件 | `SyncEvt_` + 子系统名 | `SyncEvt_Equipment`、`SyncEvt_Mail` |
| 业务管理 | 子系统名 + `Mgr` | `EquipmentMgr`、`MailMgr` |
| 调试 | `GM` | `GM.lua` |

**代码量参考**（基于 DM42 项目经验）：

| 层级 | 文件数 | 说明 |
|------|--------|------|
| 常量 | 2 | 槽位 + 同步事件，长期维护，人人可改 |
| 加密 | 1 | 密钥表 + 编解码，写好即封存，非必要不改 |
| DB 层 | 10~30 | 每加一个子系统新建一个，骨架统一，易审查 |
| 同步层 | 5~20 | 大多数子系统需要同步，简单子系统可省略 |
| Manager 层 | 10~30 | 随着业务自然增长，耦合度最高 |

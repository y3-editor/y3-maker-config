# 建造系统 (Building / Construction)

## 概述

建造系统让单位（通常是工人/农民/工程师）通过**建造技能**在地图上施工出**建筑类单位**，并支持**升级**、**取消**、**资源消耗**、**进度推进**等机制。

**典型用途**：
- RTS：农民造主城、兵营、防御塔
- 塔防：玩家放置防御塔
- 模拟经营：建造功能性建筑
- 陷阱：原地放置诱杀装置（可叠在其他单位脚下）

**与常规主动技能的差异**：
- 走专门的建造施放流程（`BUILDING_ABILITY = 4`），不使用常规 `指示器类型` 下拉
- 底层使用 `BUILD = 6` 建造指示器（不在常规指示器枚举里暴露）
- 释放后会创建一个**待建建筑**实体，按时间推进进度，完成后转为可用建筑

> ⚠️ **关键设计约束（WAR3 流派固化）**：
> 建造系统深度绑定 WAR3 三族建造模式，由 `ability_build_subtype` 枚举决定流程分支。
> Human(=2) **服务端未实现**，编辑器若选用会触发 `ERROR: on_human_construction_start, not implemented`，请勿使用。
> 详见下文「`ability_build_subtype` 详解」。

> 与之相关但**不属于建造系统**的概念：
> - **采集技能** (`PICK_ABILITY`) — 走右键采集流程，参见 `06-技能.md`
> - **科技升级** (`upgrade_tech`) — 整玩家级别的科技，不是单建筑升级
> - **生产/训练** — 由建筑出生的单位，不在本文档讨论

### 升级流程（独立于建造）

```
GameAPI.start_build_upgrade(unit, new_unit_key, keep_kv)
        │
        ▼
检查：未在建造/升级中 + 在 build_upgrade_list + 人口/资源 + pre_condition
        │
        ▼
扣升级费（按 RELATIVE_UPGRADE_COST 规则）+ 占用人口
        │
        ▼
ConstructSystem.start_upgrade(build_time)
        │
        ▼
┌──────────────────────────────────┐
│  事件: 单位-升级开始              │
│  (ET_UNIT_BUILD_UPGRADE_START)   │
│  含 old_unit_key / new_unit_key  │
└──────────────────────────────────┘
        │
        ├─ cancel_build_upgrade(return_res)
        │     │ upgrade_return_resource ← 按 UPGRADE_REFUND_RATE
        │     ▼
        │   ┌──────────────────────────────────┐
        │   │  事件: 单位-升级取消（回滚类型） │
        │   │  (ET_UNIT_BUILD_UPGRADE_CANCEL)  │
        │   └──────────────────────────────────┘
        │
        ▼
on_upgrade_finish → upgrade_to_new_unit(new_unit_key, keep_kv)
        │
        ▼
┌──────────────────────────────────┐
│  事件: 单位-升级完成              │
│  (ET_UNIT_BUILD_UPGRADE_FINISH)  │
└──────────────────────────────────┘
```

## 物编字段

### 单位侧（建筑类单位 + 工人）

**目录**：`editor_table/unit/`

| 字段 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `type` | int | — | 单位类型，建筑须为 `2 (BUILDING)` |
| `build_time` | number | — | 建造耗时（秒） |
| `build_res_cost_list` | array | `[]` | 建造资源消耗列表，元素 `[res_key, value]` 二元组，引用玩家属性（RoleResKey）；value 为 Fix32 |
| `build_upgrade_list` | array | `[]` | 可升级到的目标单位 UnitKey 列表 |
| `build_ignore_dyn_collision` | bool | false | **建造时是否忽略其他单位的动态碰撞**，陷阱类必开 |
| `show_building_attack_range` | bool | false | 建造预览阶段显示攻击范围 |
| `building_attack_range_sfx` | sfx | — | 攻击范围预览特效 |
| `pre_condition` | — | — | 训练/购买/建造该单位的前置条件 |

### 技能侧（建造技能）

**目录**：`editor_table/abilityall/`

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `ability_cast_type` | int | ✅ | 必须为 `4`（建造技能） |
| `build_list` | array | ✅ | 该技能可建造的单位 UnitKey 列表（多选） |
| `build_rotate` | number | — | 建造完成后建筑朝向角度 |
| `ability_build_subtype` | enum | — | 建造方式（WAR3 流派），见下表，**默认建议 `Default(1)`** |
| `cold_down_time` / `ability_cost` | array | — | 与常规技能一致 |

> **关键**：建造技能不需要配置 `指示器类型`，编辑器会自动走 BUILD 指示器。

### `ability_build_subtype` 详解（**重要**）

服务端枚举（`commons/black_box/consts/lua_const.py:133`）：

| 值 | 名称 | 流程行为 | 工人状态 | 推荐场景 |
|----|------|----------|----------|----------|
| `1` | **Default** | **跳过施工过程**，立即触发 START → FINISH，无 ConstructComponent | 工人不参与、不绑定 | 塔防放塔、即时陷阱、无 RTS 感的玩法 |
| `2` | ~~Human~~ | ⛔ **未实现**（源码注释掉，运行时报 `ERROR not implemented`） | — | **禁用** |
| `3` | **Orc** | 工人移动到位 → **变身建筑**（碰撞关闭、动画隐藏） → 完工后还原到附近空位 | `constructor_unit` 绑定该建筑，期间禁攻/禁动/禁技能/禁拾取 | 兽族风格 RTS |
| `4` | **Undead** | 工人放下建筑后**自由离开**，有施工进度（走 ConstructSystem） | 工人不绑定，可继续干活 | 自由度最高的 RTS 流程（与 Default 区别：Undead 有进度条，Default 无） |
| `5` | **Elf** | 工人**变身古树**，完工时**销毁工人**；取消则复原 | 同 Orc 绑定 + 占用人口 | 精灵族古树流派 |

**关键约束**：
- `Default + build_time=0` 等价，都是**直奔完成**，**无 ConstructComponent**，期间无法 `get_construction_progress` / `cancel_construction`
- **Orc/Elf/Undead + build_time=0 会降级 Default**：工人变身/绑定全部跳过，等同于直接用 Default
- `Orc`/`Elf` 期间工人挂这些状态：`BUFF_not_draw | forbid_move | forbid_attack | forbid_ability | forbid_item_use | unable_ai_select | unable_to_attacked | can_not_be_filter`（工人被完全锁死且不可被攻击）
- **待建建筑**挂：`BUFF_forbid_attack | forbid_ability | forbid_move | forbid_turn | forbid_item_use`（**不含 unable_to_attacked，可被攻击**）
- **不支持多工人协作建造同一建筑**（`constructor_unit` 是单引用）

### 全局规则字段（影响所有建造经济）

通过 `规则编辑器` 配置，影响**全图所有建造**：

| 规则 Key | 默认 | 作用 |
|---------|------|------|
| `CONSTRUCTION_REFUND_RATE` | — | 取消建造时的资源退还比例（0~1），未配置则**完全不退** |
| `CONSTRUCTION_DAMAGE_REFUND_PENALTY` | bool | 是否启用受伤罚金。开启后退还按受伤比例衰减 |
| `UPGRADE_REFUND_RATE` | — | 取消升级时的资源退还比例 |
| `UPGRADE_DAMAGE_REFUND_PENALTY` | bool | 升级版受伤罚金 |
| `RELATIVE_UPGRADE_COST` | bool | 升级费用计算方式：true=按目标完整费用；false=按目标-当前差额 |

**退还公式**（`construct_member.py:140-148`）：
```
penalty_ratio = 0.9 * progress - (hp_cur/hp_max - 0.1)
final_rate    = base_rate * (1 - max(0, penalty_ratio))
退还           = build_res_cost_list[i] * final_rate
```

## Lua API 速查

### Ability 实例 API

| 方法 | 签名 | 说明 |
|------|------|------|
| `ability:set_build_rotate(angle)` | `(number)` | 设置建造朝向（覆盖物编 `build_rotate`） |
| `ability:set_ability_build_id(unit_key)` | `(py.UnitKey)` | 动态切换该建造技能要造的单位类型 |

| `ability.handle:api_set_ability_build_area(area)` | `(py.Area)` | 限制可建造区域 |

### Unit 实例 API

| 方法 | 签名 | 说明 |
|------|------|------|
| `unit:is_building()` | `() -> boolean` | 是否为建筑类单位 |
| `unit.handle:api_create_building_on_point(build_key, point)` | `(py.UnitKey, py.Point)` | **直接发布建造命令（目标点）**。走完整 `release_skill_cmd`，**会触发 `技能-建造技能释放前` 事件**，dry_run 也能用 |

### GameAPI（建造经济 / 进度 / 升级）

```lua
-- ============ 资源消耗（物编级，所有该类型建筑共用）============
GameAPI.get_build_res_cost(unit_key, res_key)            -- -> py.Fixed
GameAPI.set_build_res_cost(unit_key, res_key, value)

-- ============ 建造时间（物编级）============
GameAPI.get_build_time(unit_key)                         -- -> py.Fixed
GameAPI.set_build_time(unit_key, build_time)

-- ============ 单建筑实例的进度与速率 ============
GameAPI.get_construction_progress(unit)                  -- -> py.Fixed (0~1)
GameAPI.set_construction_progress(unit, progress)
GameAPI.get_construction_factor(unit)                    -- -> py.Fixed
GameAPI.set_construction_factor(unit, factor)            -- 加速/减速建造

-- ============ 取消建造（待建状态）============
GameAPI.cancel_construction(unit, return_res)            -- return_res=true 退还资源

-- ============ 升级（已完成的建筑→新类型）============
GameAPI.start_build_upgrade(unit, upgrade_unit_key, keep_custom_kv_tag)
GameAPI.cancel_build_upgrade(unit, return_res)
GameAPI.get_build_upgrade_progress(unit)                 -- -> py.Fixed
GameAPI.set_build_upgrade_progress(unit, progress)
GameAPI.get_build_upgrade_factor(unit)
GameAPI.set_build_upgrade_factor(unit, factor)
GameAPI.get_build_upgrade_list(unit)                     -- -> py.Table
GameAPI.set_build_upgrade_list(unit, lua_table)          -- 动态调整可升级目标

-- ============ 建造指示器格子配置（玩家级 UI）============
-- ⚠️ y3 lualib meta 把参数标成 (RoleID, UnitGroup) 是错的！
-- 服务端实际签名：set_build_pointer_move_grids(grid_x_count, grid_y_count) 全是整数
GameAPI.set_build_pointer_move_grids(grid_x, grid_y)        -- 整数：每次移动多少格
GameAPI.set_build_pointer_move_grids_offset(offset_x, offset_y)  -- 整数：偏移多少格

-- ============ 模拟建造（用于试算/预览，不真正扣资源）============
GameAPI.set_use_skill_dry_run_state(release_id, is_dry_run)
-- 一次性标志，pop 即清；只能在「技能-建造技能释放前」事件回调里用 data.ability_release_id 调用
```

### 运行时状态查询（Python 侧实现，未导出 lualib）

`construct_member.py` 提供两个状态精准查询，比 `is_building()` 更准。**lualib 当前无包装**，可走 `unit.handle:try_get_construction_component()` / `try_get_upgrade_component()` 直接调用：

| Handle 方法 | 返回 | 说明 |
|------------|------|------|
| `try_get_construction_component()` | component / nil | 是否**正在施工**（待建状态） |
| `try_get_upgrade_component()` | component / nil | 是否**正在升级** |

> 替代写法：`GameAPI.get_construction_progress(unit) < 1` 也能判断"在施工"，进度为 1 即已完工。

## 事件参考

### 单位侧建造事件

| 事件名 | Key | 主要参数 | 说明 |
|--------|-----|----------|------|
| `单位-建造开始` | `ET_UNIT_CONSTRUCT_START` | `unit` | 待建建筑生成、施工开始 |
| `单位-建造取消` | `ET_UNIT_CONSTRUCT_CANCEL` | `unit` | 建造被取消（玩家手动 / API） |
| `单位-建造完成` | `ET_UNIT_CONSTRUCT_FINISH` | `unit` | 建筑就绪可用 |

### 单位侧升级事件（**之前文档遗漏**）

| 事件名 | Key | 主要参数 | 说明 |
|--------|-----|----------|------|
| `单位-升级开始` | `ET_UNIT_BUILD_UPGRADE_START` | `unit`, `old_unit_key`, `new_unit_key` | `start_build_upgrade` 调用成功 |
| `单位-升级取消` | `ET_UNIT_BUILD_UPGRADE_CANCEL` | `unit`, `old_unit_key`, `new_unit_key` | 升级被取消，单位回滚到 `old_unit_key` |
| `单位-升级完成` | `ET_UNIT_BUILD_UPGRADE_FINISH` | `unit`, `old_unit_key`, `new_unit_key` | 单位类型已切换到 `new_unit_key` |

### 技能侧建造事件

| 事件名 | Key | 主要参数 | 说明 |
|--------|-----|----------|------|
| `技能-建造完成` | `ET_ABILITY_BUILD_FINISH` | `ability`, `build_unit` | 通过建造技能完成时触发，能拿到造出来的建筑实例 |
| `技能-建造技能释放前` | `ET_BUILD_SKILL_BEFORE_RELEASE` | `player`, `unit`, `ability`, `new_unit_key`, `ability_target_pos`, `ability_release_id` | 命令将要发布时，可读取/响应。**dry_run 模式下也会触发** |

> `技能-建造技能释放前` 是建造系统**最关键**的拦截点：参数 `new_unit_key` 表示玩家选了哪种建筑、`ability_target_pos` 是目标点、`ability_release_id` 可与 `set_use_skill_dry_run_state` 配合做模拟试算。
>
> 官方编辑器描述（`editor_export_language_data_zh.py`）：
> > "当建造命令发布之前触发，**即使在打开技能指示器时勾选了仅模拟，也能触发该事件**"

## 典型场景

### 场景 1：RTS 农民建造主城（标准流程）

物编配置：
- 建造技能：`ability_cast_type = 4`、`build_list = [TOWNHALL_KEY]`
- 建筑单位：`type = BUILDING(2)`、`build_time = 60`、`build_res_cost_list = [{wood, 200}, {gold, 400}]`、`build_upgrade_list = [TOWNHALL_T2_KEY]`

```lua
local TOWNHALL = 134218001
local TOWNHALL_T2 = 134218002

-- 完工提示
y3.object.unit[TOWNHALL]:event('单位-建造完成', function(_, data)
    y3.game.tip(data.unit:get_owner(), '主城建造完成！')
end)

-- 升级
function upgrade_townhall(townhall)
    GameAPI.start_build_upgrade(townhall.handle, TOWNHALL_T2, true)
end
```

### 场景 2：塔防一键放塔（脚本直接发起建造，跳过技能 UI）

```lua
local WORKER = ...  -- 工程车单位
local TOWER  = 134218010

local p = y3.point.create(x, 0, z)
WORKER.handle:api_create_building_on_point(TOWER, p.handle)
```

### 场景 3：陷阱可叠放在其他单位脚下

物编：陷阱单位的 `build_ignore_dyn_collision = true`。技能照常配置。无额外代码。

### 场景 4：动态切换建造目标（同一技能可造多种建筑，按等级解锁）

```lua
y3.object.ability[MULTI_BUILD]:event('技能-建造技能释放前', function(_, data)
    local lv = data.unit:get_level()
    if lv >= 5 then
        -- ⚠️ 必须调 set_ability_build_id，改 data.new_unit_key 无效（只是事件参数快照）
        data.ability:set_ability_build_id(ADVANCED_BUILDING_KEY)
    end
end)
```

> **注意**：`data.new_unit_key` 是事件触发时 `ability.build_id` 的快照，直接赋值无效。要动态切换目标，必须调用 `ability:set_ability_build_id(K)`。

### 场景 5：付费加速建造

```lua
function rush_build(building, cost)
    local p = building:get_owner()
    if p:get_attr('gold') >= cost then
        p:add_attr('gold', -cost)
        GameAPI.set_construction_factor(building.handle, Fix32(5))  -- 5x 速度
    end
end
```

### 场景 6：取消建造退还资源

```lua
y3.object.unit[BUILDING_KEY]:event('单位-建造取消', function(_, data)
    -- 引擎默认按 cancel_construction(return_res=true) 已退；
    -- 如有自定义资源（非 res_cost_list），在此手动补退
    local owner = data.unit:get_owner()
    owner:add_attr('mana_crystal', 50)
end)

-- 主动取消（如玩家点 X）
GameAPI.cancel_construction(building.handle, true)
```

### 场景 7：建造前模拟试算（不真正扣资源）

```lua
y3.object.ability[BUILD_KEY]:event('技能-建造技能释放前', function(_, data)
    -- 试算阶段：避免重复扣资源/触发 buff
    GameAPI.set_use_skill_dry_run_state(data.ability_release_id, true)
    -- 自行做合法性判断后再决定是否真正释放
end)
```

### 场景 8：监听升级三事件（解锁科技 / 统计成就）

```lua
y3.game:event('单位-升级开始', function(_, data)
```

## 注意事项

| 项 | 说明 |
|---|---|
| 建造技能不配指示器类型 | 编辑器会自动走 BUILD 指示器，配了也无效 |
| `build_list` 与 `set_ability_build_id` | 前者是物编静态可造列表，后者是当前选中目标，运行时切换用后者 |
| `build_time = 0` 等价于 `Default subtype` | 都是直接 START → FINISH，**无 ConstructComponent**，无法 set/get progress、无法 cancel |
| `Human(=2)` 子类型禁用 | 服务端 `on_human_construction_start` print `ERROR not implemented`，建造会卡死 |
| `is_building()` 检测 | 看的是单位 `type == 2 (BUILDING)`，**不是**当前是否在施工状态 |
| 待建状态的判定 | 优先用 `unit.handle:try_get_construction_component()`；备用 `get_construction_progress < 1` |
| 升级期间单位状态 | `start_build_upgrade` 后单位进入忙碌态，常规 AI/移动指令会被拒。**但不挂任何 buff，单位可被攻击、可掉血**。升级取消退还公式按**净掉血**算（`initial_hp - hp_cur`），非绝对血量 |
| 升级取消会回滚单位类型 | 不是销毁，是回到 `old_entity_no`，但 hp_cur 保持升级中受到的伤害 |
| 资源退还需配规则 | 未配 `CONSTRUCTION_REFUND_RATE` / `UPGRADE_REFUND_RATE`，`return_res=true` 也**完全不退** |
| 受伤罚金二阶效应 | 退还公式 `penalty = 0.9*progress - (hp_cur/hp_max - 0.1)`；penalty>0 时 `rate *= 1-penalty`。满血不罚，残血（如 10%）会大幅扣减退还 |
| 陷阱叠放 | 必须 `build_ignore_dyn_collision = true`，否则会被其他单位卡位；**仅建造指示器阶段生效**（C++ collision_box 消费）。完工后碰撞取决于 `dynamic_collision_r`，若需完工后无碰撞应配 `dynamic_collision_r = 0`。完工瞬间脚下有单位时**不会主动挤开**，可能穿模叠加 |
| 不支持多工人协作建造 | `constructor_unit` 是单引用，第二个工人发命令会冲掉第一个。**冲突后果**：第一个工人状态异常（buff 残留、动画卡住），需手动清理 |
| 模拟建造 (`set_use_skill_dry_run_state`) | 一次性 token，服务端读取后立即 pop，无需手动 false。只能在 `技能-建造技能释放前` 事件中用 `data.ability_release_id` 调用。**set true 后**：不扣资源、不创建建筑、不发后续 START/FINISH 事件。该事件在 dry_run 模式下也会触发（可用于预览/试算） |
| 升级费用计算 | 受 `RELATIVE_UPGRADE_COST` 规则影响：true=按目标建造完整费用；false=按目标-当前差额 |
| 升级失败可能补退人口 | 升级开始就占用新单位人口，失败/取消会还原旧人口 |
| `set_build_pointer_move_grids` 类型标错 | y3 lualib meta 写 `(RoleID, UnitGroup)` 是错的，**实际是 `(int, int)` 格子数** |
| 建造预览特效仅本地 | `show_building_attack_range` / `building_attack_range_sfx` 走 `@visual_call`，敌方看不见 |

## 与其他系统的协作

- **资源系统（玩家属性）**：`build_res_cost_list` 引用 `py.RoleResKey`，与 `06-技能.md` 中的 `ability_cost`（蓝量）解耦；扣资源逻辑见 `_deduct_resource`
- **人口系统**：建造和升级都会经过 `population_occupation` 校验，扣不出人口直接报 `$Nofood` 中断
- **规则系统**：5 个 RULE 字段直接控制经济行为，配规则比写 Lua 更标准
- **科技/前置条件**：`pre_condition` 字段挂前置（如要求已存在某建筑、达成某科技），由 `check_precondition` 统一校验
- **AI 行为树**：施工中 / 升级中单位会被打上 `BUFF_unable_ai_select` 等状态，AI 自动不选它
- **UI**：建造按钮使用官方组件 type:17 技能栏即可承载，自动接收冷却/资源不足等状态
- **Buff**：施工中单位被强制挂上 `UNDER_CONSTRUCTION_BUFF_STATUS`（禁攻禁动等），完工时移除
- **多人同步**：核心建造命令在 `must_sync.lua` 登记，业务层只需正常调用、勿放本地客户端独占代码块

# 地图逻辑资源（点 / 路径 / 区域）落盘机制

## 结论

Y3 逻辑资源（点、路径、区域）有两种来源：

- **静态资源**：编辑器中预先放置，保存在 `logicres.json`，通过 `res_id` 获取
- **动态资源**：运行时代码创建，有 `res_id`（运行时递增分配），但不持久化，关卡重启后消失  - **路径和区域**：创建时注册进 `logic_res_mgr`，可通过 `get_xxx_by_res_id` 取回
  - **点**：仅是坐标封装（`Fix32Vec3`），未注册，不能通过 `get_point_by_res_id` 取回
### 通过 res_id 获取资源

以下接口均可取回静态资源；其中路径和区域的动态对象同样可取回（它们在创建时均会注册进 `logic_res_mgr`）。

```lua
-- 点（仅静态，动态创建的点是坐标封装，无法通过此接口取回）
y3.point.get_point_by_res_id(res_id)

-- 路径（静态 & 动态）
y3.road.get_road_by_res_id(res_id)

-- 区域（静态 & 动态，按形状分）
y3.area.get_circle_by_res_id(res_id)     -- 圆形区域
y3.area.get_rectangle_by_res_id(res_id)  -- 矩形区域
y3.area.get_polygon_by_res_id(res_id)    -- 多边形区域
y3.area.get_by_res_id(res_id)            -- 自动探测形状
```

### 动态资源：运行时创建

```lua
-- 点
y3.point.create(x, y, z)

-- 路径（以起点创建，再用 road:add_point 追加）
y3.road.create_path(start_point)

-- 区域
y3.area.create_circle_area(point, radius)
y3.area.create_rectangle_area(point, horizontal_length, vertical_length)
y3.area.create_rectangle_area_from_two_points(point_one, point_two)
y3.area.create_polygon_area_by_points(point1, point2, ...)
```

静态资源真正依赖的主数据文件是：

- `maps/<地图名>/logicres.json`

仅有编辑器资源树索引文件中的 ID，不足以让运行时成功取到对象。

---

## 三层落盘结构

当在编辑器中放置逻辑资源后，工程中会同时更新三类文件：

### 1. 主数据：`logicres.json`

这是**权威数据源**。

- 点资源按类型 `1024` 分组
- 矩形区域按类型 `2048` 分组
- 圆形区域按类型 `4096` 分组
- 路径资源按类型 `8192` 分组
- 多边形区域按类型 `4194304` 分组

### 点资源结构示意

```json
"1024": {
  "10000": {
    "id": 10000,
    "name": "点",
    "placed_lock": false,
    "placed_visibility": true,
    "pos": {
      "__tuple__": true,
      "items": [...]
    },
    "tags": {
      "__tuple__": true,
      "items": []
    },
    "type": 1024
  }
}
```

关键字段：
- `id`
- `type = 1024`
- `pos`

### 路径资源结构示意

```json
"8192": {
  "10005": {
    "id": 10005,
    "name": "路径",
    "road_point_list": [...],
    "tags": {
      "__tuple__": true,
      "items": []
    }
  }
}
```

关键字段：
- `id`
- `type = 8192`（在子点中体现）
- `road_point_list`

### 矩形区域资源结构示意

```json
"2048": {
  "10010": {
    "id": 10010,
    "name": "矩形区域",
    "type": 2048,
    "pos": { "__tuple__": true, "items": [...] },
    "width": 500.0,
    "height": 500.0,
    "weather": null,
    "sound": null,
    "color": null,
    "placed_visibility": true,
    "placed_lock": false,
    "tags": { "__tuple__": true, "items": [] }
  }
}
```

关键字段：
- `id`
- `type = 2048`
- `pos`（左下角坐标）
- `width` / `height`

### 圆形区域资源结构示意

```json
"4096": {
  "10011": {
    "id": 10011,
    "name": "圆形区域",
    "type": 4096,
    "pos": { "__tuple__": true, "items": [...] },
    "radius": 300.0,
    "weather": null,
    "sound": null,
    "color": null,
    "placed_visibility": true,
    "placed_lock": false,
    "tags": { "__tuple__": true, "items": [] }
  }
}
```

关键字段：
- `id`
- `type = 4096`
- `pos`（圆心坐标）
- `radius`

### 多边形区域资源结构示意

```json
"4194304": {
  "10012": {
    "id": 10012,
    "name": "多边形区域",
    "type": 4194304,
    "road_point_list": [...],
    "weather": null,
    "sound": null,
    "color": null,
    "placed_visibility": true,
    "placed_lock": false,
    "tags": { "__tuple__": true, "items": [] }
  }
}
```

关键字段：
- `id`
- `type = 4194304`（在子点中体现）
- `road_point_list`（各顶点坐标列表，结构与路径相同）

---

### 2. 已放置资源索引：`editor/existingmodelinfo.json`

记录“编辑器中已经放置过哪些逻辑资源”。

结构示意：

```json
[10000, "点", 1024]
[10005, "路径", 8192]
```

作用：
- 编辑器快速遍历
- 已放置资源列表索引

不是运行时点坐标的权威来源。

---

### 3. 编辑器目录索引：`editor/folderinfo/folderinfo_editor_resource.json`

记录资源 ID 挂在哪个编辑器文件夹树位置。

结构示意：

```json
"10000": ["某个资源路径", 0]
```

作用：
- 编辑器资源树显示
- 目录挂载关系

注意：
- 这个文件**不包含点的空间坐标本体**
- 仅看到某个 ID 出现在这里，不能说明运行时一定能取到点

---

## 为什么仅有 folderinfo 索引不够

`folderinfo_editor_resource.json` 只描述：
- 资源 ID
- 资源所属目录
- 目录中的排序索引

它不描述：
- 点坐标
- 路径控制点
- 资源完整属性

而 `y3.point.get_point_by_res_id(id)` 需要引擎从实际逻辑资源数据中构造 Point 对象，因此真正依赖的是：

- `logicres.json["1024"][id]`

如果只有 folderinfo 记录，没有 `logicres.json` 中的有效点数据，运行时仍会报：

```text
EAError: ea_point_is_none
```

## 类型号速记

| 类型号 | 名称 | `LogicUnitType` 常量 |
|---|---|---|
| `1024` | 点 | `POINT = gen_2_power(10)` |
| `2048` | 矩形区域 | `AREA_RECTANGLE = gen_2_power(11)` |
| `4096` | 圆形区域 | `AREA_CIRCULAR = gen_2_power(12)` |
| `8192` | 路径 | `ROAD_POINT_LIST = gen_2_power(13)` |
| `4194304` | 多边形区域 | `AREA_POLY = gen_2_power(22)` |

---

## 正确判断一个逻辑点是否可被运行时读取

不要只检查：
- `folderinfo_editor_resource.json` 中是否有该 ID

而应优先检查：
- `logicres.json` 中是否存在该 ID
- 是否位于点类型分组 `1024` 下
- 是否具备有效 `pos`

也可以再辅助检查：
- `editor/existingmodelinfo.json` 中是否登记
- `editor/folderinfo/folderinfo_editor_resource.json` 中是否挂到资源树

---

## 静态资源 vs 动态资源

| | 静态资源 | 动态资源 |
|---|---|---|
| 创建方式 | 编辑器放置 | 运行时代码创建 |
| 数据来源 | `logicres.json` | 仅存在于内存 |
| 有 `id` | ✅ | ✅（运行时分配） |
| `id` 持久化 | ✅ | ❌ |
| 关卡重启后保留 | ✅ | ❌ |
| 获取方式 | `get_xxx_by_res_id` | 创建时返回对象引用；**区域和路径**亦可持有其 `res_id` 后通过 `get_xxx_by_res_id` 取回 |

> 动态资源的 `id` 由引擎在当前静态最大 ID 基础上递增分配（`cur_max_id += 1`），仅存活于本次运行时。重启后该 ID 失效。
>
> 注意：动态**区域**和动态**路径**在创建时均调用 `logic_res_mgr.create_logic_res_obj` 注册进资源管理器，因此持有 `res_id` 后可通过 `get_xxx_by_res_id` 重新取回同一对象。动态**点**（`y3.point.create`）仅是 `Fix32Vec3` 坐标封装，未经过 `create_logic_res_obj`，无法通过 `get_point_by_res_id` 取回。

## 区域的 Lua API

区域按形状分为三种，获取方式如下：

```lua
-- 明确知道形状时，推荐用具体接口
local circle = y3.area.get_circle_by_res_id(res_id)
local rect   = y3.area.get_rectangle_by_res_id(res_id)
local poly   = y3.area.get_polygon_by_res_id(res_id)

-- 不确定形状时，可省略 shape 参数，自动探测（按圆→矩形→多边形顺序尝试）
local area = y3.area.get_by_res_id(res_id)
```

`Area.SHAPE` 枚举：

```lua
Area.SHAPE = {
    CIRCLE    = 1,  -- 圆形
    RECTANGLE = 2,  -- 矩形
    POLYGON   = 3,  -- 多边形
}
```

> 静态区域保存在 `logicres.json`，若 `logicres.json` 中无对应 ID，运行时将抛错。动态区域则直接存在于 `logic_res_mgr` 内存中，不依赖 `logicres.json`。

---

## 推荐排查顺序

当出现：

```text
EAError: ea_point_is_none
```

建议按以下顺序检查：

1. `logicres.json` 中是否存在该 ID
2. 该 ID 是否位于点类型 `1024` 下
3. 该点是否有有效 `pos`
4. `existingmodelinfo.json` 中是否登记
5. `folderinfo_editor_resource.json` 中是否存在对应索引
6. 地图是否已正确保存并重新加载

---

## 适用结论

以下接口的行为因类型而异：

```lua
-- 仅静态资源：y3.point.create 只是坐标封装（Fix32Vec3），未注册到 logic_res_mgr，无法通过此接口取回
y3.point.get_point_by_res_id(id)

-- 静态 & 动态均可：动态路径和区域创建时均调用 create_logic_res_obj 注册进资源管理器，可用此接口取回
y3.road.get_road_by_res_id(id)
y3.area.get_circle_by_res_id(id)
y3.area.get_rectangle_by_res_id(id)
y3.area.get_polygon_by_res_id(id)
```

对于**静态资源**，其主数据来源是 `logicres.json`，仅凭编辑器 folderinfo 索引不足以让运行时成功取到对象。

对于**动态路径和动态区域**，引擎在创建时通过 `logic_res_mgr.create_logic_res_obj` 分配运行时 `res_id` 并注册，因此 `get_xxx_by_res_id` 同样能取到该对象，但 `res_id` 不持久化，关卡重启后失效。

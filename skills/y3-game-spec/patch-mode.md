# Patch Mode：增量迭代流程

> **Patch Mode** 适用于对现有工程的所有修改/优化/扩展场景。
> 无论改动大小，都需要走完整的「增量策划案 → 增量执行案 → 增量测试案」闭环。

## 触发条件

用户需求匹配以下模式之一：
- "修改XX"、"修 Bug"、"加个功能"、"调整数值"
- "只改 UI"、"只改 Lua"、"只改物编"
- "优化XX"、"迭代XX"
- "添加新功能"、"丰富内容"、"增加系统"、"扩展玩法"

## Step 1：需求分析 + 范围识别

```
用户请求
    │
    ▼ 检查是否存在主干文档
    检查 spec-config.json 的 gameName
    │
    ▼ 如有主干策划案/执行案，读取获取上下文
    {paths.designDoc}/{gameName}设计案.md
    │
    ▼ 分析改动范围
    ├─ 涉及哪些模块？（UI/Lua/物编）
    ├─ 新增还是修改？
    ├─ 影响范围多大？
    └─ 与现有功能的依赖关系？
    │
    ▼ 输出范围分析
    "本次改动的范围是：{范围描述}"
```

**输出**：确定改动范围 + 功能边界

---

## Step 2：创建增量目录

```
目录命名规则：
{paths.patches}/{date}-{brief}/

示例：
openspec/patches/2026-04-23-事件系统升级/
openspec/patches/2026-04-24-修复技能Bug/
```

**目录结构**：
```
openspec/patches/2026-04-23-事件系统升级/
├── 增量策划案.md      # Step 3 生成
├── 增量执行案.md      # Step 5 生成
└── 增量测试案.md      # Step 7 生成
```

> 配置路径：在 `spec-config.json` 中可通过 `paths.patches` 自定义

## Step 4：用户确认方案（强制）

> ⚠️ **此步骤不可跳过**

```
生成增量策划案后
    │
    ▼ 向用户展示方案
    "以上是本次改动的设计方案，请确认是否可以开始实现？"
    │
    ▼ 使用 ask_user_question 获得确认
    ├─ 用户确认 → 继续 Step 5
    └─ 用户要求调整 → 修改策划案 → 重新确认
```

**禁止行为**：
- ❌ 用户说"开始实现"就直接写代码
- ❌ 跳过确认步骤
- ❌ 自行判断"不需要确认"

**例外情况**：
- ✅ 用户明确说"不用确认直接做"

## Step 6：制作执行

按增量执行案的 Phase 顺序执行制作。

**强制检查**：

| 检查项 | 规则 | 触发时机 |
|--------|------|---------|
| API 预检 | 验证 API/事件存在性 | Lua 代码编写前 |
| 热更保存 | hotfix → 等 3 秒 → save | 物编/UI 变更后 |
| Lua 审查 | 执行 y3-lua-review | Lua 修改完成后 |

**路由顺序（Mixed 模式）**：
```
物编（数据层）
    ↓
UI（表现层）
    ↓
Lua（逻辑层）
```

## Step 8：执行测试

使用 `y3-auto-test` 执行增量测试案。

```
1. 启动游戏
2. 执行测试用例 PT-01 ~ PT-xx
3. 记录测试结果
4. 如失败：修复后重新验证（最多 2 轮）
```

## Patch Mode 规则总结

| 规则 | 说明 |
|------|------|
| 无论改动大小 | 都必须走完整流程 |
| 完整三件套 | 必须生成增量策划案/执行案/测试案 |
| 用户确认 | Step 4 必须获得用户明确确认才能开始实现 |
| 强制检查不简化 | API 预检、热更保存、Lua 审查与 Full Mode 一致 |
| 永久保留历史 | 增量目录及文档永久保留 |
| ID 前缀区分 | 功能使用 `P-` 前缀，测试用例使用 `PT-` 前缀 |

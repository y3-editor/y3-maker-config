# 策划案生成 — 多轮引导详细指南

> 本文件是 `y3-game-spec` 阶段 1 的详细执行指南。
> 被 `SKILL.md` 引用，激活阶段 1 时必须读取本文件。

---

## 🔄 引导流程总览

```
Step 1 游戏类型       ← 确定品类和核心定位
    │
    ▼
Step 1.5 参考游戏边界校验 ⭐🆕  ← 用户提到具体游戏名时立即查 feasibility-redlines.md
    │
    ▼
Step 2 核心玩法       ← 明确玩什么、怎么玩 + 即时边界审查
Step 3 角色与单位     ← 玩家控制谁、对抗谁 + 即时边界审查
Step 4 战斗与技能     ← 战斗方式、技能设计 + 即时边界审查
Step 5 经济与成长     ← 资源获取/消耗、升级体系 + 即时边界审查
Step 6 UI 与交互      ← 需要哪些界面、操作方式 + 即时边界审查
Step 7 胜负与流程     ← 怎么赢/怎么输、关卡/波次流程 + 即时边界审查
Step 8 数值与平衡     ← 关键数值框架
    │
    ▼
Step 8.5 视听与交互细节澄清 ← 兜底扫描（特效/音效/镜头/飘字/UI手感），智能跳过
    │
    ▼
Step 9 跨 Step 冲突检测 + 最终复审 ⭐🆕  ← 不再做全量审查（已实时完成）
    │
    ▼
汇总生成策划案
    │
    ▼
Step 10 策划案结构完整性校验 ⭐🆕  ← 仅查结构/章节/表格骨架，不审具体内容
    │
    ▼
用户审阅确认 → 进入阶段 2
```

> ⚠️ **每一步都必须使用 `ask_user_question` 与用户交互**，等用户回答后再进入下一步。
> 用户可随时说"跳过"来使用默认值，AI 会基于前面的回答自动补全。
> 🆕 **每一步收到用户回答后，必须立即对照 `feasibility-redlines.md` 做实时审查**，命中边界立刻打断处理，禁止积累到 Step 9。
> 🎯 **品类先验**：进入 Step 2 前，AI 必须根据 Step 1 选择的游戏类型，参照下方「游戏类型引导差异表」对后续 Step 的选项进行动态调整。

## 🔁 实时边界审查模板（Step 1.5 + Step 2~7 共用）

> 本模板由 Step 1.5、Step 2~7 共同复用，**禁止跳过**。
> 边界清单文件：`feasibility-redlines.md`

### 执行步骤

```
Step N 用户回答 = ask_user_question(...)
    │
    ▼
1. 关键词解析
   - 用户答案中是否提到具体游戏名（"像XX"、"参考XX"）？
   - 是否包含可触发边界的特征词（"第三人称"、"FPS"、"横版"、"3D 自由探索"、
     "准星瞄准"、"千人战场"、"光线追踪"等）？
    │
    ▼
2. 对照边界
   - 命中游戏名 → 查 feasibility-redlines.md「参考游戏命中表」
   - 命中特征词 → 查 feasibility-redlines.md「特征边界表」
    │
    ▼
3. 分级处理
   ├─ ❌ 硬边界 → 立即用 ask_user_question 打断，按 redlines.md 标准模板提问，
   │              用户重选方案后才能进入下一 Step
   ├─ ⚠️ 降级项 → 立即用 ask_user_question 告知降级影响，记录决策后进入下一 Step
   └─ ✅ 未命中 → 直接进入下一 Step
```

### 关键纪律

| 纪律 | 说明 |
|------|------|
| **不允许凑活** | ❌ 边界一律驳回，禁止说"虽然不太适合但勉强能做" |
| **必须打断** | 命中 ❌ 时不能"先记下来到 Step 9 再说"，必须当下处理 |
| **必须给替代方案** | 驳回时至少提供 2 个 Y3 可行的替代方向 |
| **必须告知降级影响** | ⚠️ 项必须明确说"原版长什么样 vs 在 Y3 上会变成什么样" |
| **替代方案被拒后允许终止** | 用户可选"我坚持原方向不在 Y3 上做"，礼貌结束流程，不强行继续 |

## Step 1：游戏类型与核心定位

**目的**：确定游戏品类，后续所有引导内容会基于此调整。

**使用 `ask_user_question` 提问**：

```
问题: 你想做什么类型的游戏？
选项:
  - 塔防 — 建造防御塔阻挡敌人波次
  - RPG — 角色扮演，成长+战斗+剧情
  - 肉鸽/Roguelike — 随机关卡+永久死亡+构建流派
  - 生存 — 收集资源、抵御威胁、活下去
  - MOBA — 多人对战，推塔/团战
  - 休闲/解谜 — 消消乐、连连看、关卡解谜
  - 竞速/动作 — 跑酷、竞速、动作挑战
  - [其他] （用户自定义输入）
```

**AI 后续动作**：
- 记录 `game_type`
- **🆕 进入 Step 1.5 参考游戏边界校验**

## Step 1.5：参考游戏边界校验（🆕 强制前置审查）

> **触发时机**：Step 1 完成后立即执行。
> **必读文件**：`feasibility-redlines.md`

### 执行步骤

**Step 1.5.1: 提取参考游戏 / 关键词**

从 Step 1 的回答中提取：
1. 用户是否在 [其他] 中输入了具体游戏名（如"像兽人必须死"、"参考王者荣耀"）？
2. 用户的描述是否包含视角/操作类边界特征词（如"第三人称"、"FPS"、"横版"）？

**Step 1.5.2: 对照边界清单**

按上文「🔁 实时边界审查模板」执行：
- 命中"参考游戏命中表"中的 ❌ → 走"硬边界打断流程"
- 命中"特征边界表"中的 ❌ → 走"硬边界打断流程"
- 命中 ⚠️ → 走"降级告知流程"
- 未命中 → 直接进入 Step 2

**Step 1.5.3: 硬边界打断流程**

**使用 `ask_user_question` 提问**（按 `feasibility-redlines.md` 标准模板）：

```
问题: ⛔ 检测到你的需求触碰 Y3 引擎边界，无法实现：

  ❌ [具体需求] — 原因：[边界说明，例如「Y3 是 2.5D 俯视角引擎，不支持第三人称跟随相机」]

  Y3 是 2.5D 俯视角引擎，建议改为以下方向：

选项:
  - [替代方案 A]（推荐，最接近原始需求）
  - [替代方案 B]（保留部分核心体验）
  - [替代方案 C]（完全换方向）
  - 我坚持原始需求，不在 Y3 上做了（终止流程）
```

**Step 1.5.4: 用户回答处理**

- 选择替代方案 → 用替代方案重置 `game_type` / `core_gameplay`，进入 Step 2
- 选择终止 → 礼貌结束流程，不强行继续

### AI 后续动作
- 边界触发与处理结果记录到 `redline_history`，最终写入策划案第 10 章「技术可行性说明」
- 如果用户选了替代方案，后续 Step 2-8 的提问基于替代方案而非原始需求

| 类型 | Step 3 侧重 | Step 4 侧重 | Step 5 侧重 | Step 7 侧重 |
|------|-------------|-------------|-------------|-------------|

## Step 2：核心玩法描述

**使用 `ask_user_question` 提问**：

```

（AI 根据 Step 1 的类型给出参考示例）

选项（根据 game_type 动态生成）:
  例如塔防类:
  - 在固定路线上建造防御塔，抵挡越来越强的怪物波次

  - 选择英雄，击败波次怪物，收集装备变强
  - 探索地图，完成任务，击败 Boss

**使用 `ask_user_question` 提问**：

```

  - 键盘操作（WASD 移动 + 技能键）
  - [其他]
```

**追问**（技能数量）：

```
问题: 每个英雄/塔需要几个主动技能？
选项:
  - 1 个（简单明了）
  - 2-3 个（适中复杂度）（Recommended）
  - 4 个（MOBA 风格，QWER）
  - 不需要主动技能（纯自动/纯被动）
```

**追问**（Buff/特殊效果）：

```
问题: 需要哪些特殊效果/Buff？（可多选）
选项:
  - 减速
  - 中毒/持续伤害
  - 眩晕/定身
  - 护盾/加血
  - 增伤/增速
  - 不需要特殊效果
  - [其他]
```

**追问**（技能视听反馈，🆕 关键细节）：

```
问题: 技能释放时的视听反馈风格？（影响后续特效/音效配置）
选项:
  - 标准反馈（每个技能有施法特效 + 命中特效 + 命中音效）（Recommended）
  - 强化反馈（额外加镜头震屏 / 击中顿帧 / 大招专属运镜）
  - 简洁反馈（只有命中特效，无音效，适合休闲/原型）
  - 让 AI 按技能类型自动决定（控制类配音效，AOE 配震屏，等等）
  - [其他]（自定义说明，例如"只有大招要震屏"）
```

**追问**（伤害数字飘字）：

```
问题: 战斗中是否需要伤害数字飘字？
选项:
  - 需要，普通伤害白色，暴击红色加大（Recommended）
  - 需要，所有伤害统一样式
  - 不需要（界面更简洁）
  - [其他]
```

**AI 后续动作**：
- 记录 `combat_style`、`skill_count`、`buffs`

## Step 6：UI 与交互设计

**目的**：确定需要哪些界面和操作方式。

**使用 `ask_user_question` 提问**（可多选）：

```
问题: 需要哪些游戏界面？（可多选）
选项:
  - 开始游戏界面（主菜单：开始游戏 / 选关 / 查看存档 / 设置 / 退出）
  - 英雄选择界面（游戏开始前选角色）
  - 战斗 HUD（血条/蓝条/等级/金币等常驻信息）
  - 技能栏（显示可用技能）
  - 物品栏/背包（显示持有物品）
  - 商店面板（购买装备/道具）
  - 波次/关卡信息栏（当前波次、倒计时等）
  - 结算面板（胜利/失败结算）
  - 小地图
  - [其他]
```

**追问**（如果选了"开始游戏界面"，🆕）：

```
问题: 开始游戏界面需要包含哪些子功能？（可多选）
选项:
  - 开始游戏（直接进入第一关 / 当前进度）（Recommended）
  - 选关（关卡列表，已通关/未解锁状态显示）
  - 存档管理（查看/读取/删除存档）
  - 设置（音量、画质、按键等）
  - 退出游戏
  - 排行榜 / 成就
  - [其他]
```

**追问**（操作方式）：

```
问题: 玩家通过什么方式操作？
选项:
  - 鼠标点选+右键移动（RTS 风格，Y3 引擎默认支持）（Recommended）
  - WASD 键盘移动
  - 纯点击 UI（放置/卡牌类）
  - 键盘快捷键（数字键选择，字母键释放技能）
  - [其他]
```

**⚠️ 自动检测：内置 3C 冲突**（🆕 关键规则）

Y3 引擎内置了一套**默认 3C（Camera / Character / Control）**：
- 鼠标左键点选单位
- 鼠标右键发出移动/攻击命令
- 鼠标拖拽框选
- 鼠标中键/滚轮控制相机

当用户选择**非 RTS 风格**操作方式时，必须**显式禁用**冲突的默认行为，否则会出现"按 W 想前进，结果右键点了一下角色又跑回原地"之类的体验灾难。

**触发条件检查**：
| 用户在 Step 4 / Step 6 的选择 | 是否需要禁用默认 3C | 禁用范围 |
|------|------|------|
| Step 4 选「键盘操作 (WASD)」 | ✅ 必须禁用 | 右键移动、左键点选移动 |
| Step 4 选「回合制」 | ✅ 必须禁用 | 右键移动、自动攻击命令 |
| Step 4 选「纯自动战斗」 | ✅ 必须禁用 | 右键移动、左键点选（避免误操作） |
| Step 6 选「纯点击 UI（放置/卡牌）」 | ✅ 必须禁用 | 右键移动、世界点选 |
| Step 6 选「WASD 键盘移动」 | ✅ 必须禁用 | 右键移动 |
| Step 6 选「鼠标点选+右键移动（RTS）」 | ❌ 保持默认 | — |

**如果命中触发条件**，**使用 `ask_user_question` 提问**：

```
问题: 检测到你的操作方式与 Y3 默认 RTS 操作冲突，需要禁用部分默认行为以避免手感问题。
请确认禁用范围：（推荐采用 AI 自动方案）

选项:
  - ✅ 采用 AI 推荐的禁用方案（自动屏蔽冲突项，保留相机控制）（Recommended）
  - 全部禁用（连相机控制也接管，自己写一套完整 3C）
  - 部分保留（我手动指定保留哪些默认行为）
  - 暂时不禁用（接受可能的体验问题）
  - [其他]
```

**AI 后续动作**：
- 记录 `disable_default_3c` 决策清单（如 `disable_right_click_move=true`, `disable_left_click_select=true`, `keep_camera_control=true`）
- 决策结果写入策划案 **6.4 节「默认 3C 禁用清单」**
- 该清单会在 Phase 3 Lua 编写时由 `y3-lua-pipeline` 强制读取，调用对应 API 禁用默认行为

**追问**（按钮反馈，🆕 关键细节）：

```
问题: 按钮点击时希望的反馈表现？

### Step 10.1：不完整时的补齐路由（强制使用 `ask_user_question`）

发现结构问题后**必须**立刻询问用户如何处理，不得擅自决定：

```
问题: 策划案结构校验发现 N 项结构问题（详见上方校验报告），
     这些只是"骨架/结构"层面的问题，不涉及内容合理性。
     你希望怎么处理？

选项:
  - 🤖 让 AI 自动补齐结构（仅填充骨架与"无 / 不适用"标注，不新增实质内容）（Recommended）
  - ✍️ 我自己修补（暂停流程，我编辑好文件后再回来继续）
  - 👀 先看看完整报告再决定（AI 重新详细展示每一项问题）
  - ⏭️ 忽略结构问题强行进入下一步（不推荐，会影响后续一致性校验）
```

### 各选项的后续动作

| 用户选择 | AI 动作 |
|---------|---------|
| 🤖 自动补齐 | 按补齐规则修改文件 → 重新跑一次 Step 10 校验 → 仍不通过最多重试 2 次（共 3 轮）→ 第 3 轮失败必须**触发"硬切换"**（见下方） |
| ✍️ 自己修补 | 输出待修补清单 + 文件路径 → 暂停流程并等待用户说"修好了" → 用户回来后重新跑 Step 10 |
| 👀 查看详细报告 | 按章节展开列出每一项缺失，然后再次触发本追问 |
| ⏭️ 忽略强行继续 | 在策划案末尾追加一条醒目的 `> ⚠️ 结构校验被用户跳过，待补项：[清单]` → 进入用户审阅确认，并在进入阶段 2 时提示 `doc-consistency` 校验可能失败 |

### 🆕 自动补齐 3 次失败后的"硬切换"路径（强制）

当 🤖 自动补齐第 3 轮仍失败时，**禁止再尝试自动补齐**，必须立即用 `ask_user_question` 让用户**逐项**指明处理方式：

```
问题: AI 自动补齐已尝试 3 次仍未通过结构校验，剩余 N 项缺失需要你逐项确认。

剩余缺失清单：
  [1] 6.4 默认 3C 禁用清单 — 表格未生成
  [2] 8.2 怪物属性 — 第 8/9/10 波缺数据
  [3] 11.3 镜头与震屏 — 占位符未替换
  [...]

请选择整体处理策略：
选项:
  - 🔧 我逐项告诉你每项怎么处理（AI 会按列表依次追问每一项）（Recommended）
  - ✍️ 我自己手动修文件（暂停流程，等我说"修好了"再回来）
  - ⏭️ 强行跳过所有剩余项（每项追加 `> ⚠️ 结构校验未通过，待补` 标记，进入阶段 2 时 doc-consistency 必失败，需人工豁免）
  - 🔄 推翻重来（回到 Step 1）
```

**逐项追问模式**（用户选 🔧 时）：

对剩余每一项，依次用 `ask_user_question` 提问：

```
问题: [项目 N: 章节标题] 该如何处理？

选项:
  - 显式标注"无 / 不适用"（推荐 80% 的场景）（Recommended）
  - 我现在口述内容（AI 写入）
  - 暂时跳过该项（追加 ⚠️ 标记）
```

**硬切换的纪律**：

| 纪律 | 说明 |
|------|------|
| **禁止再次自动补齐** | 第 3 轮失败后，AI 不可再以"我再试一次"为由跳过用户确认 |
| **禁止合并追问** | 每项必须独立 `ask_user_question`，不可"打包让用户一次性回答" |
| **逐项决策必须落盘** | 每收到一项回答，立即更新策划案文件，再问下一项 |
| **完成后重跑校验** | 所有项处理完后再跑一次 Step 10 校验，目标是判定为通过；若仍存在结构问题（极小概率），降级为"强行跳过"并明确警告用户 |

### 自动补齐规则（🤖 选项专用）

AI 在执行自动补齐时**必须严格遵守**以下原则，禁止越界：

| ✅ 允许 | ❌ 禁止 |
|--------|---------|
| 补齐缺失的章节/小节标题 | 编造具体数值（HP / 伤害 / 金币数等） |
| 将空壳表格显式标注为"无 / 不适用（原因：[基于 Step 1~9 的既有记录]）" | 虚构单位、技能、Buff 等条目 |
| 清理残留的 `[Step N 数据]` 占位符（替换为"无"或合理默认） | 变更用户在 Step 1~9 已确认的任何决策 |
| 补齐 ASCII 流程图/布局图等纯结构性内容 | 新增未经用户确认的玩法/机制 |
| 修正编号不连续问题 | 修改已填写的内容（即使看起来不合理） |

> ⚠️ 所有自动补齐的条目必须追加标注 ` <!-- 自动补齐 -->`，方便用户在审阅阶段快速识别。

| **数值可调** | 数值只是初始框架，在阶段 2/3 中可以随时调整 |
| **确认才进入阶段 2** | 策划案必须经用户明确确认后才能进入下一阶段 |

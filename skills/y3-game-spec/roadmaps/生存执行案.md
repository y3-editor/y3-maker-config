### �️ 地形策略

| 策略 | 说明 |
|------|------|
| **地形来源** | 默认使用模板 `roguelike-survival` 中的防守图地形，不另行生成或修改 |
| **地形尺寸** | 64×64（点阵 65×65），与模板一致 |
| **装饰物** | 沿用模板自带装饰物（14 个已随物编导入），不增不减 |
| **纹理/植被** | 沿用模板原始纹理与植被配置 |
| **刷怪点/出生点** | 使用模板地形中预设的逻辑节点位置（通过 `get_point_by_res_id` 引用），坐标由模板决定 |

### 🔖 逻辑节点命名规则（强制）

> 所有通过 MCP 创建（`add_point` / `add_point_path`）或模板继承的逻辑节点，名称必须可读且与功能对应。

| 规则 | 说明 | 示例 |
|------|------|------|
| **语义化命名** | 名称直接描述节点在游戏中的功能含义 | `小怪进攻路线`、`英雄出生点`、`Boss刷怪区`、`装备掉落区` |
| **中文优先** | 优先使用中文命名，便于团队成员理解 | `小怪进攻路线` 而非 `monster_route_01` |
| **禁止无意义名称** | 严禁使用流水号、随机字符串等无法直接识别的命名 | `point_001`、`path_a`、`node_3` |
| **路径节点** | 路径用 `add_point_path` 创建时，路径名应描述该路线的用途 | `小怪进攻路线`、`Boss进攻路线`、`巡逻路线` |
| **代码引用** | Lua 中通过 `y3.scene.get_point_by_res_id()` 按名称/ID 获取节点，名称即为 `res_id` | — |

> 原则：逻辑节点命名 = 自文档化。任何人读节点名称即能理解其功能，无需查代码。

> 原则：地形 = 模板原样，不做任何地形编辑操作（`terrain_*` MCP 调用全部跳过）。执行案仅负责物编 + UI + Lua 逻辑，地形完全由模板提供。

### �🔒 模板使用强制约束

> 复现工程时禁止 AI 自由发挥地形/装饰物布局，必须严格沿用入选模板的原始配置。详细约束项参考塔防执行案 Phase 1 同名章节。

## 🔍 API 预检（Step 5）

> 关键 API / 事件结论表：

| API / 事件 | 结论 | 备注 |
|------------|------|------|
| `y3.player(id):create_unit(unit_id, point, facing)` | 可用 | 实例方法，非 `y3.player.create_unit` |
| `y3.point.create(x, y, z)` | 可用 | 非 `create_point` |
| `unit:set_attr(attr, val, type)` | 可用 | type 取 `'基础'/'加成'/...`；不要用于改 `'移动速度'` 做长期调整（破坏动画） |
| `unit:get_hp() / get_mp()` | 可用 | 当前生命/魔法值 |
| `unit:get_attr(attr)` | 可用 | 读最终属性，如 `'最大生命'/'最大魔法'/'移动速度'` |
| `unit:reborn(point)` | 可用 | 复活，非 `revive` |
| `unit:damage{target=, type=, damage=, common_attack=}` | 可用 | **表参数**，非位置参数 |
| `unit:add_buff{key=, source=, time=, stacks=}` | 可用 | 同上 |
| `unit:has_modifier(buff_key)` | 可用 | 检测 Buff，非 `has_buff` |
| `unit:attack_target(target, 0)` | 可用 | 持续追击 |
| `ability:get_owner()` | 可用 | 在 `单位-施放技能` 事件中取 caster |
| `ability:get_key()` | 可用 | 取技能 ID |
| `y3.point.get_point_by_res_id(res_id)` | 可用 | 取场景静态点 |
| `y3.game:event('单位-施放技能', cb)` | 可用 | **主动技能用这个**，不要用 `施法-完成`（仅持续/通道触发） |
| `y3.game:event('单位-死亡', cb)` | 可用 | `data.unit` 为死亡单位，`data.killer` 为击杀者 |
| `y3.ctimer.loop / wait` / `y3.ltimer.loop / wait` | 可用 | 计时器 |
| `y3.ui.get_ui(player, path)` | 路径不存在时 **throw 而非 nil**，必须 pcall 包裹 |
| `ui:bind_unit_attr('当前值', '生命', 0)` | 进度条初始不同步 | 改手动 `set_current/max_progress_bar_value` 定时刷新 |
| `ui:set_skill_on_ui_comp(ability)` | 可用 | type:17 技能按钮绑定 |
| `ui:set_ui_unit_slot(unit, slot_type, idx)` | 可用 | type:20 物品栏绑定 |
| `y3.game.exit()` / `GameAPI.quit_game()` | 不存在 | 改为隐藏 UI / 设置 game_over 展示结算页 |
| `y3.const.UIAttr` 常量名 | 是 `'当前值'/'最大值'/'文本'`，**不是** `'进度'/'进度最大值'` |
| `y3.const.UnitAttr` 常量名 | 是 `'生命'/'魔法'/'最大生命'/'最大魔法'`，**不是** `'生命值'/'魔法值'` |

## 🪤 物编关键陷阱（实战沉淀）

| 陷阱 | 现象 | 正确做法 |
|------|------|----------|
| **`ori_speed` ×100 存储** | 编辑器值 280 → 引擎内部 28000 → 单位瞬移；动画速率正常但视觉飞行 | 编辑器填 `期望cm/s ÷ 100`（英雄~2.8、怪物 1.4~3.0） |
| **运行时改移速破坏动画** | `set_attr('移动速度', X)` 改速度但物编不变，动画 `current/(ori_speed×100)` 失配 | 长期速度调整改物编；临时加/减速可接受短暂不匹配 |
| **`ability_damage_range` 与 AOE 实际半径解耦** | 物编 damage_range 仅决定指示器/施法范围，Lua AOE 命中范围在脚本里独立判定 | 在 Lua 集中维护 `SKILL_EFFECT[skill_id].radius`，按地图尺寸调整（中心防守 4 角刷怪场景建议 600+） |
| **`ability_cast_type` 影响事件触发** | `cast_type=2`（点击释放）走 `单位-施放技能`；持续/通道技能才走 `施法-完成` | 主动伤害/位移技能统一监听 `单位-施放技能` |
| **UI 路径必须含中间 layout** | `StatusBar.label_gold` 报不存在，实际为 `StatusBar.status_bar.label_gold` | UI 生成后必跑 `gen_ui_tree.py`，按节点树写完整路径 |

## 📐 场景尺寸 × 技能半径联动校验（实战沉淀）

| 场景类型 | 中心→边缘距离 | 推荐 AOE 半径 |
|----------|---------------|---------------|
| 紧凑（32×32） | ~1500 | 200~400 |
| 中等（64×64） | ~2500~3000 | 400~800 |
| 大型（128×128） | ~5000+ | 800~1500 |

> 本工程 64×64，中心防守 + 四角刷怪 → AOE 600+ 才能覆盖到边缘怪物。Lua 写技能效果配置表时必须根据策划场景规格定半径，不可照搬物编小数值。

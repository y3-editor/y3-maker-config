# Y3 地形常识库

> 本文件记录 Y3 编辑器地形系统的基础常识，用于约束 AI 识图时的判断范围。
> **持续积累**：每发现一条新的识别误差，在此补充对应常识。

## ⛔ 首要原则

**本文件只提供约束范围和可行做法，不定义"XX = 几层悬崖"这种固定公式。**
具体参数（几层悬崖、几格宽的水体等）永远由图片实际内容决定，常识只告诉 AI "在 Y3 里可以怎么做"以及"不能怎么做"。

## 高度体系

Y3 有两层高度系统：

| 层级 | 名称 | 变化量 | 效果 | 用途 |
|------|------|--------|------|------|
| **整数高度**（terrain_height） | 悬崖级高度 | 每级 ±2（API height 值） | 产生悬崖/深水边界 | 区域间层级分隔、山峰 |
| **微调高度**（detail_height） | 隆起/降低 | < 0.1 | 平滑地形起伏，无悬崖 | 偶尔微调地面，不要大面积使用 |

| 常识 | 说明 |
|------|------|
| **基础高度 = 0** | Y3 地图默认地形高度为 0，不需要刷就是这个值；识图时各区域无特殊说明则高度为 0，不需要主动抬高 |
| **1 层悬崖 = terrain_height +2** | API 层面：抬高 1 层悬崖需传 `height=2`（即 `target_height * 2`）；`terrain_get_block` 回读的 `terrain_height` 值也是 2 的倍数。例：抬高 1 层→height=2，2 层→height=4，降低 1 层→height=-2 |
| **同一平坦区域高度是一个固定整数** | 不能给"高度 -2~0"这种范围！同一片平坦区域的高度就是一个整数（如 0），不存在内部基准波动 |
| **高度是相对的** | 地形高度表达的是相对关系，不是绝对世界坐标；各区域高度基准相同时，全部写 0 即可 |
| **微调高度偶尔使用** | 隆起/降低操作（detail_height < 0.1）仅用于非常微小的地形起伏，不要大面积使用，否则表现不好 |

## 水体体系

Y3 有三种水体类型，视觉和通行性完全不同：

| 水体类型 | 挖掘深度 | 可通行 | 有悬崖边缘 | 视觉效果 | 类比 |
|---------|----------|--------|-----------|---------|------|
| **深水（deep）** | 2 层 | ❌ 不可通行 | ✅ 有 | 深蓝色，不透明 | 深湖、深河 |
| **浅水（shallow）** | 1 层 | ✅ 可通行 | ✅ 有 | 浅蓝色，半透明 | 浅河道，有河岸 |
| **平面水（plain）** | < 0.1 | ✅ 可通行 | ❌ 无 | 几乎透明，直接铺在地面 | 雨后积水、水洼 |

| 常识 | 说明 |
|------|------|
| **深水固定挖 2 层** | 引擎从陆地高度往下挖 2 层悬崖高度，水面与陆地齐平。如陆地高度=0，则水底=-2 |
| **浅水固定挖 1 层** | 引擎从陆地高度往下挖 1 层悬崖高度，水面与陆地齐平。如陆地高度=0，则水底=-1 |
| **浅水有悬崖但可通行** | 浅水虽然可以走，但周围仍有悬崖边缘（像河道），地形上有凹陷 |
| **平面水无悬崖** | 平面水直接铺在地面上，不改变地形形状，没有悬崖边缘，像地面积水 |
| **平面水不挖地形** | 平面水只铺水面纹理，不改变地形高度 |
| **bottom_height = 水底绝对高度** | 刷深水/浅水时传入的 bottom_height 是水底的绝对高度值。如陆地高度=0，深水 bottom_height=0 表示从高度 0 处开始挖（引擎自动挖 2 层到 -2） |
| **水体升降关系** | 深水区域用"升高地形"刷一下 → 变浅水（从挖2层变为挖1层）；浅水区域再升高一下 → 变陆地（不再挖掘） |

### 河流水体类型判断规则

图中蓝色区域（河流）**不能全部判断为深水**，需根据宽度和上下文区分：

| 判断条件 | 水体类型 | 说明 |
|---------|---------|------|
| 有桥跨越的河段 | 深水（deep） | 桥的存在说明水深到需要桥才能通行 |
| 无桥且很宽（≥3格）的大面积水体 | 深水（deep） | 大河/湖泊，不可通行 |
| 无桥但稍有宽度（1~2格）的河段 | 浅水（shallow）+ 两侧斜坡 | 浅河道，单位可沿斜坡走下去趟水再走上来 |
| 无桥且特别细/窄的水流（<1格） | 平面水（plain） | 小溪/水沟，直接走过去 |

**浅水+斜坡组合的结构**（河道截面）：
```
地面(0)  斜坡  浅水区  斜坡  地面(0)
────────╲      ░░░      ╱────────
         ╲(-1) ░░░(-1) ╱
          ╲────░░░─────╱
```
即：地面高度 0 → 斜坡下到 -1 → 浅水区在 -1（可通行）→ 斜坡上回 0

## 山峰体系

| 常识 | 说明 |
|------|------|
| **山峰 = 悬崖叠加 + 山顶摆件** | 山峰可以通过多层悬崖叠出地形高度，然后在最高层放置山峰/岩石类装饰摆件来表现尖峰效果。具体几层悬崖取决于图片中山峰的明度渐变层次 |
| **山峰是渐变区域，不是平台** | 山峰的高度从山脚到峰顶逐层递增（每层 +1），不是直接跳到某个固定值的平台 |
| **山峰地形高差不超过 +5** | 相对周围平地，山峰地形部分高差不超过 +5；Y3 引擎中 +5 已经是视觉上非常高的山峰，实际图片中通常只需要 2~3 层 |
| **山峰不是 cliff** | cliff 是裂缝（地形空洞），山峰图案是 ground 类型的局部高地 |

## cliff（裂缝）体系

| 常识 | 说明 |
|------|------|
| **cliff ≠ 悬崖/高山** | Y3 中 `cliff` 专指裂缝（terrain_height = -40），是地形空洞，单位不可通行 |
| **cliff 只在明确标注时使用** | 图片中只有明确画出裂缝/空洞区域时才识别为 cliff，山峰、高地、悬崖边缘均不是 cliff |
| **cliff 通过专用接口写入** | 必须使用 `terrain_set_crack_block` 接口，不能用高度接口写 -40 |

## 颜色与地形的对应关系

颜色有两个独立维度，分别对应不同的地形信息：

| 颜色维度 | 对应信息 | 作用范围 |
|---------|---------|---------|
| **色相（Hue）** | 纹理类型（草地/泥土/岩石等） | 跨区域有效 |
| **明度（Brightness）** | 区域内局部高度（亮=高，暗=低） | **仅在同一色相区域内有效** |

| 常识 | 说明 |
|------|------|
| **色相 = 纹理** | 黄绿色相=草地、棕色相=泥土、灰色相=岩石——色相只决定纹理类型，与高度无关 |
| **明度 = 区域内高度** | 同一色相区域内，明度越高（越亮）= 局部越高（如山峰处），明度越低（越暗）= 局部越低 |
| **⛔ 禁止跨色相比较明度来判断高度** | 黄绿色天然比灰色"亮"，但这是色相本身的特性，不代表草地区域比岩石区域更高！不同色相之间的明度差异不能用来推断高度关系 |
| **所有区域默认等高（基准=0）** | Y3 地图中 99% 的情况各区域基准高度相同（都是 0），除非文字**明确写出**某区域高度不同（如"左下低洼区，高度-1"） |
| **桥梁 = 等高佐证 + 摆件** | 图中桥梁/金色方块连接两个区域 → 证明这些区域等高。桥本身是摆件（不是地形），Pass 3 时在桥的位置放置桥梁摆件 |
| **图片文字"颜色深浅代表高度"的正确理解** | 图片中如果写了"颜色深浅代表高度"，在 Y3 语境下的正确含义是：**同一色相区域内**的明度变化代表高度变化，**不是全图所有颜色的整体亮暗比较**。不能按字面意思跨色相应用 |
| **文字说明需通过 Y3 常识重新解读** | 图片文字的优先级高于颜色推断，但文字本身也需要用 Y3 地形常识来理解，不能脱离 Y3 语境按字面意思理解 |
| **白色线条 = 道路** | 图中白色线条代表道路，不参与地形写入（地形类型仍为 ground），道路感通过摆件实现 |

### 纹理区域一致性

| 常识 | 说明 |
|------|------|
| **一片区域纹理大概率一致** | Y3 中一片连续区域通常使用同一种纹理，不会在同一区域内随意混搭多种纹理 |
| **特殊情况：刻意铺设异种纹理** | 少数情况下会在某种纹理区域上刻意覆盖另一种纹理（如在精灵草地上铺一片亡灵腐地），这属于特殊设计 |

### 纹理颜色映射参考

| 图片中的颜色 | 对应纹理 |
|------------|---------|
| 棕色 | 泥土（mud） |
| 黄绿色 | 草地（grass） |
| 灰色 | 岩石（rock） |
| 沙黄色 | 沙地（sand） |
| 白灰色 | 雪地（snow） |

## 装饰物 vs 植被系统

Y3 有两套完全独立的地表物件系统：

| 维度 | 装饰物系统（entity_create） | 植被系统（vegetation_draw） |
|------|--------------------------|--------------------------|
| **本质** | 独立的 3D 模型实体 | 地表贴片（billboard/面片） |
| **用途** | 树木、桥梁、岩石、房屋、山峰 | 草、花、芦苇等地表点缀 |
| **ID 类型** | model_id（编辑器模型 ID） | vegetation_type（植被类型 ID） |
| **粒度** | 单个实体，精确放置 | 区域铺设，有密度参数 |
| **坐标** | 世界坐标 [x, y, z] | 植被网格坐标（与地形网格不同！） |
| **可旋转/缩放** | ✅ 是 | ❌ 否（仅随机参数） |

| 常识 | 说明 |
|------|------|
| **树木是装饰物不是植被** | 树木通过 entity_create 放置 3D 模型，不走植被系统 |
| **草/花是植被不是装饰物** | 草丛、花朵通过 vegetation_draw 铺设，不需要逐个放置 |
| **植被坐标系 ≠ 地形坐标系** | 植被用 `calculate_foliage_grid_pos`，地形用 `calculate_terrain_grid_pos` |
| **资源 ID 映射表** | 装饰物 model_id 和植被 vegetation_type 的完整映射见 `references/decoration-model-ids.md` |

## 纹理面板限制

| 常识 | 说明 |
|------|------|
| **地图纹理面板** | 每个地图有一个纹理面板，只有面板中启用的纹理才能通过 MCP 写入 |
| **添加纹理后必须保存** | 用户添加新纹理到面板后，必须 **Ctrl+S 保存地图**，否则 MCP 调不到该纹理 |
| **170 种纹理可选** | 完整映射表见 `references/texture-ids.md`，AI 应选择具体的纹理 ID 而非笼统分类 |

## 道路体系

| 常识 | 说明 |
|------|------|
| **Y3 没有道路实体** | Y3 引擎不存在"道路"这种地形类型或对象，没有专门的道路刷写接口 |
| **道路通过摆件表现** | 道路效果是通过摆放路面摆件来逻辑上表现"这里是一条路"，本质上还是 ground 地形 + 摆件装饰 |
| **道路的纹理和旁边一致** | 道路区域的地面纹理与相邻区域相同（如旁边是草地则纹理也是草地），道路不是一种纹理类型 |
| **图片中的白色线条（道路）** | 识图时道路区域的地形类型仍然是 ground（高度跟随所在区域），纹理跟随相邻区域，"道路感"通过 Pass 3 摆件放置实现 |

## 斜坡体系

| 常识 | 说明 |
|------|------|
| **斜坡 = 高差为 1 的可通行区域** | 相邻格子高差为 1 时形成斜坡（类似 Dota 高地），需要通过 MCP 接口主动刷写 |
| **斜坡需要主动识别和生成** | 识图时需要识别出存在高低过渡的区域，并在点阵中标注斜坡；不能只标注 ground 高度然后指望引擎自动生成 |
| **斜坡自动生成规则待确认** | 引擎内部确实存在一套斜坡自动修正规则（`calculate_valid_flat`），但具体触发条件尚未完全确认，暂不依赖自动生成 |
| **⚠️ 斜坡必须最后刷** | 斜坡优先级最低，任何其他地形操作（水体、悬崖、纹理等）都会联带覆盖掉已有的斜坡。刷写顺序必须是：地形高度 → 水体 → 纹理 → **最后刷斜坡** |
| **浅水河道需要斜坡配合** | 浅水+斜坡组合用于可通行的河道：地面(0) → 斜坡下到(-1) → 浅水区(-1) → 斜坡上回(0)。斜坡在浅水刷完后最后补 |

## 坐标系

Y3 是 2.5D 编辑器，存在两套坐标系。**所有地形操作使用格点坐标系，装饰物放置使用世界坐标系。**

### 世界坐标系 (World Coordinates)

| 属性 | 说明 |
|------|------|
| **原点** | 地图中心 (0, 0, 0) |
| **x 轴** | 向右增大（正方向 = 右/东） |
| **z 轴** | 向下增大（正方向 = 下/南，俯视图） |
| **y 轴** | 表示高度（向上增大） |
| **数值类型** | 连续浮点数 |

```
            z 小（上/北）
              ▲
              │
              │
x 小 ◄────────┼────────► x 大（右/东）
（左/西）      │ (0,0,0)
              │
              ▼
            z 大（下/南）
```

### 格点坐标系 (Grid Coordinates)

| 属性 | 说明 |
|------|------|
| **原点** | 左上角 (0, 0) |
| **x 轴** | 向右增大 |
| **z 轴** | 向下增大 |
| **数值类型** | 非负整数 |
| **步长** | 1 格 = 世界坐标 2 个单位 |
| **范围** | (0, 0) ~ (W-1, H-1)，W/H 由 `get_map_info` 返回（格点数 = terrain.json 中格子数 + 1） |

```
  (0,0) ──────────────────► x 增大
    │
    │      ((W-1)/2, (H-1)/2)
    │      = 世界原点 (0, 0)
    │
    ▼
  z 增大
```

### 格点 ↔ 世界坐标互转公式

来源：`TerrainHelper.py :: calculate_terrain_grid_pos()`

```
# 格点 → 世界（W, H = get_map_info 返回的格点尺寸）
world_x = grid_x * 2 - (W - 1)
world_z = grid_z * 2 - (H - 1)

# 世界 → 格点
grid_x = (world_x + (W - 1)) / 2
grid_z = (world_z + (H - 1)) / 2
```

示例（get_map_info 返回 W=65, H=65 的地图）：

| 格点坐标 | 世界坐标 | 位置 |
|---------|---------|------|
| (0, 0) | (-64, -64) | 左上角 |
| (32, 32) | (0, 0) | 地图中心 |
| (64, 64) | (64, 64) | 右下角 |

### 图片 → 格点映射

| 属性 | 说明 |
|------|------|
| **方向** | 图片像素 (px, py) 与格点 (x, z) 方向完全一致 |
| **原点** | 图片左上角 (0, 0) = 格点 (0, 0) |
| **⛔ 翻转** | **不需要 Y 轴翻转！** 图片 y=0 在顶部，格点 z=0 也在顶部 |
| **CSV 对应** | `cluster_grid.csv` 的 grid[z][x] 直接对应格点 (x, z) |

```
图片像素空间         格点坐标系          地形结果
(0,0)──►px         (0,0)──►x         左上角 = (0,0)
  │                  │
  │ 方向一致          │ 一一对应
  │ 无需翻转          │ 无需翻转
  ▼py                ▼z
```

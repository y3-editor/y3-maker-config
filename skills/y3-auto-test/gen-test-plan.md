# 测试计划生成规范

> 当需要生成测试计划时，AI 按本文档的三步流程执行。
> 输出文件路径：`<workspace>/skills/y3-auto-test/test-plan/{游戏名称}-测试计划-{YYYY-MM-DD-HH-mm}.md`

---

## 第一步：阅读策划案

读取策划案文档，提取以下信息：

- 游戏名称（用于文件命名）
- 游戏概述（一句话描述）
- 策划案文件名（用于元信息记录）

---

## 第二步：分析功能与测试点

将策划案中的功能按以下维度分类，并为每个功能列出测试点和对应的测试方法。

### 功能分类

| 编号 | 分类 | 说明 |
|------|------|------|
| 2.1 | UI 操作 | 所有与 UI 控件交互相关的功能（按钮、面板、输入框等） |
| 2.2 | 游戏主循环 | 核心游戏逻辑（初始化、胜负判定、刷怪、回合、计分等） |
| 2.N | （可扩展） | 如存档、音效、网络等，根据策划案实际内容新增分类 |

### 测试方法（M1-M4）

> 为每个测试点从下表选择最合适的方法。如四种方法均无法满足，在「测试方法」列填写 ⚠️ 提示。

| 编号 | 方法名称 | 使用工具 | 适用场景 |
|------|---------|---------|---------|
| M1 | UI 点击触发 | `trigger_ui_touch_event_by_path`（y3runtime MCP） | UI 按钮/控件点击后的响应验证 |
| M2 | 执行 Lua | `execute_lua`（y3helper MCP） | 游戏逻辑、数值、状态、对象存在性验证 |
| M3 | 截图验证 | `capture_screenshot`（y3helper MCP） | 视觉效果、UI 渲染正确性存证 |
| M4 | 查看日志 | `get_logs`（y3helper MCP） | 事件触发、逻辑执行路径、错误信息验证 |

> 后续如有新增测试方法，在此表追加 M5、M6...

**无法覆盖时的提示规则**：若某测试点超出 M1-M4 能力范围，在「测试方法」列填写：
```
⚠️ 当前工具链无法自动化，建议人工验证或补充新方法
```

---

## 第三步：输出测试计划

### 输出规范

- **目录**：`<workspace>/skills/y3-auto-test/test-plan/`（不存在时先创建）
- **文件名**：`{游戏名称}-测试计划-{YYYY-MM-DD-HH-mm}.md`
- **禁止写入**：Step 0 清理流程、回归流程、报告模板、护盾注入、冒烟规则等执行阶段内容

### 输出文件结构模板

```markdown
# {游戏名称} 测试计划

- 生成时间：{YYYY-MM-DD HH:mm}
- 策划案来源：{策划案文件名}
- 游戏概述：{一句话描述}
```
